# 2026-05-22 变更记录

## 一、QQ 文件下载修复

**问题**：用户发送 PDF/Word 等非图片文件时，bot 始终提示"文件下载失败"。

**根因**：NapCat（llbot）的 `get_file` API 默认只返回容器内路径（`/root/xxx.pdf`），不返回 URL 或文件内容。该路径位于容器内且文件并不实际存在于宿主机，`docker cp` / 路径重映射均无效。

**修复**：
- 在 `xiaomeng-napcat` 容器内的 `config_3668605379.json` 中设置 `"enableLocalFile2Url": true`
- 此后 `get_file` 响应会包含 `base64` 字段（文件的 base64 内容）
- `core/qq/gateway.py` → `_download_qq_file()` 优先读 `base64` 字段直接写文件，移除了 6 次 docker cp 重试循环

**注意**：若容器被重建（`docker rm` 后重新 `docker run`），需重新写入此配置。

## 二、定时 AI 指令（`instruct` 任务类型）

**背景**：scheduler 原来只能定时发固定消息（`cron`/`interval` 类型）。用户希望 scheduler 能触发小萌执行动态 AI 任务（出题、搜索、生成内容等）并把结果发到群里。

**实现**：

### 1. Bot 中继服务新增 `/instruct` 接口
`core/qq/gateway.py` → `_start_relay_server()`：

```
POST http://127.0.0.1:3003/instruct
{
  "prompt": "请出一道高中数学题",
  "target": {"group_id": 123456},
  "session_key": "group:123456"  // 可选
}
```

收到请求后，以 **ADMIN 权限**将 prompt 存入对话历史，走完整 LLM 推理流程（含工具调用），把 AI 回复发到目标群/私聊。

### 2. `_make_task_coro_factory` 新增 `forced_level` 参数
允许定时任务以指定权限级别运行，不依赖 `sender_qq` 推断。

### 3. Scheduler daemon 新增 `instruct` 类型
`data/skills/scheduler/main.py`：

```json
{
  "id": "daily-quiz",
  "type": "instruct",
  "cron": "9:00",
  "prompt": "请出一道数据结构与算法中等难度题目...",
  "target": {"group_id": 1092678483}
}
```

同时删除了已有的 `quiz` 类型（含内联 DeepSeek API 调用，逻辑重复且绕过 bot 权限体系）。

### 4. `tasks.json` 更新
两个 DSA 出题任务从 `cron`（固定占位消息）改为 `instruct`（小萌真正动态出题/解析）：
- `dsa_morning_question`：09:00，让小萌出一道 DSA 题
- `dsa_evening_solution`：20:00，让小萌发今天题目的解析

### 5. 文档更新
- `data/plugins/scheduler/SKILL.md`：新增 `instruct` 类型说明，删除 `quiz` 类型
- `docs/architecture.md`：更新第五节（中继链路图），第六节（任务类型表），第八节（文件下载说明）

## 三、公式渲染修复

**问题 1**：多行连等式渲染成多张图片。
**修复**：更新 `data/skills/formula-render/SKILL.md`，加了 ⚠️ 警告和 ❌/✅ 示例，要求多行推导放进一个 `$$\begin{aligned}...\end{aligned}$$`。

**问题 2**：`\begin{aligned}` 渲染出乱码。
**根因**：`_render_usetex()` 把含 `\begin{}` 的公式包进 `$...$`（行内数学模式），LaTeX 不允许在行内模式里用 `aligned` 环境，降级到 mathtext 后也无法解析。
**修复**：`\begin{` 开头的公式改用 `\[...\]`（display math 模式）。

## 四、send_file 修复

**问题**：bot 日志显示发送成功，QQ 端实际收不到文件。
**根因**：`file://` 路径是宿主机路径，NapCat 容器内找不到对应文件。
**修复**：`_send_file()` 改为读文件内容以 `base64://` 格式上传，并增加 retcode 检查。

## 五、Docker 化部署

**背景**：send_file 排查发现 bot（systemd 裸跑）与 NapCat（Docker）路径隔离问题。决定将 bot 也容器化，统一用 docker-compose 管理。

**新增文件**：
- `XiaoMeng/Dockerfile`：基于 `python:3.13-slim`，安装 LaTeX、ffmpeg、Python 包
- `XiaoMeng/.dockerignore`：排除 `data/`（由 volume 挂载）和缓存文件
- `XiaoMeng/entrypoint.sh`：容器启动时先清理残留 PID，后台启动 scheduler，前台运行 bot
- `docker-compose.yml`（在 `/home/qwq/zcx_ai_group_friend/`）：统一管理三个容器

**设计要点**：bot 容器使用 `network_mode: host`，可直接访问宿主机上所有 127.0.0.1 服务。

**切换步骤**（待执行）：
```bash
sudo systemctl stop xiaomeng-bot && sudo systemctl disable xiaomeng-bot
cd /home/qwq/napcat_xiaomeng && docker compose down
cd /home/qwq/zcx_ai_group_friend && docker compose up -d
```
